一、系统概述与角色权限体系
-------------

### 1.1 角色体系总览

系统共设置 5 种角色，权限范围按数据粒度从省→市→网点→个人逐级收窄。

| 角色标识           | 角色名称  | 数据可见范围  | 核心权限说明 |
| -------------- | ----- | ------- | ---------------------------------- |
| ROLE_PROVINCE  | 省分管理员 | 全江西省    | 所有功能全量读写；唯一可配置佣金规则的角色              |
| ROLE_CITY      | 地市管理员 | 本地市数据   | 管理本市网点及发展人；多级审核权限                  |
| ROLE_OUTLET    | 网点管理员 | 本网点数据   | 登记发展记录；查看本网点佣金；管理本网点发展人            |
| ROLE_DEVELOPER | 发展人   | 个人数据    | 提交发展记录；查看本人佣金明细                    |
| ROLE_ADMIN     | 超级管理员 | 全局（系统级） | 账号管理、角色分配、系统初始化                    |

- 每次审核操作（通过或拒绝）自动记录一条日志
- 发展人申请审核：`target_type='DEVELOPER_APPLY'`，`audit_phase` 为 NULL
- 订单审核：`target_type='ORDER'`，`audit_phase` 为 'LEAD' 或 'FORMAL'
- 发展人申请列表和详情接口的 `auditLog` / `auditHistory` 字段从此表实时查询
- 订单列表和详情接口的 `leadAuditLog` / `formalAuditLog` 字段从此表实时查询

三、核心业务逻辑修复
----------

### 3.1 账号创建逻辑修复【问题9：安全初始密码】

**原设计问题**: 密码=手机号后6位（不安全，容易被爆破）

**改进方案**:
    1. 发展人申请审核通过时，系统生成随机密码（12位，包含大小写+数字+特殊符号）
    2. 密码保存在 sys_user.password_hash 中（BCrypt加密）
    3. 通过短信网关发送初始密码给申请人手机号
    4. 用户首次登录后，强制修改密码

### 3.2 多级审核流程【问题8修复】

**原设计**: 简单的 PENDING → APPROVED → REJECTED

**新设计**: 分级审核流程
    发展人申请（biz_developer_apply）:
      PENDING 
      → OUTLET_APPROVED（网点管理员第一级审核，可选）
      → CITY_APPROVED（地市管理员第二级审核）
      → PROVINCE_APPROVED（省级最终审核）
      ╰→ REJECTED（任何一级拒绝即终止）

    业务订单审核（biz_development_order）:
      PENDING（待审）
      → OUTLET_APPROVED（网点管理员确认，仅记录）
      → CITY_APPROVED（地市管理员审核）
      → PROVINCE_APPROVED（省级最终审核，触发佣金生成）
      ╰→ REJECTED（任何一级拒绝即终止）

### 3.3 发展人去重规则【问题7修复】

**新增验证逻辑（审核时执行）**:
    -- 检查：是否存在30天内相同客户+业务类型的有效记录
    SELECT * FROM biz_development_order 
    WHERE customer_phone = ?
      AND business_type = ?
      AND outlet_id = ?
      AND lead_status IN ('PENDING_AUDIT', 'APPROVED')
      AND DATE_SUB(CURDATE(), INTERVAL 30 DAY) <= DATE(created_at)
      AND is_deleted = 0;

    -- 如果存在，则拒绝新申请（提示：该客户在30天内已有相同业务发展记录）

### 3.4 区域合规性校验【问题4修复】

**申请审核时**:
    // 获取申请人信息
    const apply = await getApply(applyId);

    // 获取申请的网点信息
    const outlet = await getOutlet(apply.outlet_id);

    // 校验：申请的city_id + district_id 是否与网点一致
    if (apply.city_id !== outlet.city_id || apply.district_id !== outlet.district_id) {
      throw new BizException("申请区域与网点区域不匹配，请联系管理员");
    }

### 4.2 修改的接口

#### 发展人申请接口（支持多级审核）

    PUT /api/v1/developer-applies/{id}/audit
    
    请求体:
    {
      "status": "CITY_APPROVED",  // 下一步状态
      "reviewRemark": "同意通过，待省级审核",
      "approveToNextLevel": true   // 是否推送到下一级
    }
    
    说明：
    - 网点管理员审核通过后状态变为 OUTLET_APPROVED，并自动转推到地市
    - 地市管理员审核通过后状态变为 CITY_APPROVED，并自动转推到省级
    - 省级审核通过后状态变为 PROVINCE_APPROVED，自动创建sys_user和发展人账号
    - 任何人拒绝则状态变为 REJECTED，流程终止

#### 业务订单审核接口（支持多级审核）

    PUT /api/v1/orders/{id}/lead-audit
    
    请求体:
    {
      "auditLevel": "CITY",  // OUTLET | CITY | PROVINCE
      "status": "APPROVED",  // APPROVED | REJECTED
      "auditRemark": "同意，质量良好"
    }
    
    说明：
    - 网点审核通过：lead_status = OUTLET_APPROVED，转推到地市
    - 地市审核通过：lead_status = CITY_APPROVED，转推到省级
    - 省级审核通过：lead_status = PROVINCE_APPROVED，触发佣金生成

#### 发展人列表查询（添加去重检查）

    GET /api/v1/orders?customer_phone=13900001111&business_type=MOBILE_CARD&days=30
    
    响应体添加字段:
    {
      "isDuplicate": true/false,  // 是否存在30天内重复客户记录
      "duplicateOrderId": 123,    // 重复订单ID
      "duplicateMessage": "该客户在30天内已有该业务发展记录"
    }

#### 密码初始化接口【问题9修复】

    POST /api/v1/auth/init-password/{userId}
    
    说明：
    - 仅在发展人申请审核通过时调用
    - 系统生成随机密码（12位，大小写+数字+特殊符号）
    - 返回的密码仅展示一次，之后不可见
    - 通过短信网关发送密码给用户
    
    响应体:
    {
      "userId": 123,
      "username": "13900001111",
      "tempPassword": "aB@12345XyZ9",
      "smsStatus": "sent"  // sent | pending | failed
    }
